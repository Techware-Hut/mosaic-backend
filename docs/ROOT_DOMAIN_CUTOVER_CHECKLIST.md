# Root Domain Cutover Checklist

Updated: 2026-06-24

This checklist is names-only and value-safe. It records human configuration work required before promoting the backend after the canonical marketplace frontend changes from `app.mosaicbizhub.com` to `mosaicbizhub.com`.

## Release Gate

- Do not merge or deploy a `staging -> main` release until the frontend and backend root-domain correction PRs are merged to their integration branches.
- Existing release PRs that contain the superseded app-domain plan must wait for this correction.
- No wildcard CORS, arbitrary return URLs, redirect-validation removal, payment logic changes, webhook changes, or auth middleware order changes.

## GoDaddy / DNS

- Connect `mosaicbizhub.com` to the rebuilt Vercel frontend project using Vercel/GoDaddy supplied record targets.
- Configure `www.mosaicbizhub.com` as a redirecting alias to the apex while preserving path and query string.
- Decide the approved transition behavior for `app.mosaicbizhub.com`; keep it temporarily only until apex authenticated smoke passes.
- Preserve `api.mosaicbizhub.com` pointing to the backend.
- Do not delete unrelated TXT, MX, DKIM, SPF, DMARC, verification, or mail records.

## Vercel

- Apex is attached to the frontend project and set as primary.
- `www` redirects to apex.
- Vercel launch hostname remains available for QA/preview.
- No legacy project claims apex or `www`.
- Frontend env var names to reconcile: `NEXT_PUBLIC_APP_URL`, `NEXT_PUBLIC_CLIENT_BASE_URL`, `NEXT_PUBLIC_API_BASE_URL`, public Sentry env names if used.

## AWS Elastic Beanstalk

- Env var names to reconcile: `FRONTEND_URL`, `CORS_ORIGINS`, `API_BASE_URL`, `COOKIE_DOMAIN`, `CONNECT_RETURN_URL`, `CONNECT_REFRESH_URL`, `CONNECT_RETURN_PATH`, `CONNECT_REFRESH_PATH`, `BILLING_PORTAL_RETURN_URL`, `SENTRY_ENVIRONMENT`, `SENTRY_RELEASE`.
- Do not automatically change `COOKIE_DOMAIN`; first confirm whether cookies are host-only or intentionally shared across `mosaicbizhub.com` subdomains.
- Preserve HttpOnly auth cookies, Secure in production, SameSite behavior, credentialed requests, and explicit CORS origins.

## Stripe

- Checkout return URLs should land on apex paths.
- Connect return and refresh URLs should land on apex paths.
- Billing portal return URL should land on an apex path.
- Subscription success/cancel URLs should land on apex paths.
- Verify allowlisted production domains or return URLs in Stripe Dashboard without changing payment/webhook behavior.

## Google OAuth

- Backend callback URI remains on `https://api.mosaicbizhub.com`.
- Frontend post-auth redirect should resolve to apex.
- Authorized JavaScript origins, if configured, should include apex and approved temporary QA origins only.
- Redirect sanitization must reject arbitrary origins.

## Email Links

Audit generated links for verification, password reset, welcome email, vendor application status, order confirmation, booking/order notifications, invitations, and referrals. Links generated by backend helpers should resolve to apex unless explicitly API-owned.

## Sentry

- Keep environment and release reporting intact.
- Reconcile hostname-specific alert filters or release dashboards after apex cutover.

## Post-Cutover Smoke

1. Apex frontend loads rebuilt marketplace.
2. `www` redirects to the equivalent apex path.
3. Old `app` subdomain follows approved transition behavior.
4. API health and readiness remain on `https://api.mosaicbizhub.com`.
5. CORS preflight from apex succeeds with credentials.
6. CORS preflight from unapproved origin fails.
7. Customer login, business-owner login, and logout work.
8. Google OAuth returns to apex.
9. Verification and email links return to apex.
10. Stripe checkout, Connect, billing portal, and subscription returns land on apex paths.
11. Vendor onboarding submission still works.
12. Marketplace browsing calls API subdomain.
13. Featured products use `GET /api/featured-products`; no code uses `/api/products/featured`.

## Rollback

- Revert backend code to the previous release branch commit if URL validation breaks auth/payment flows.
- Restore prior EB env values only if the cutover config caused the incident.
- Re-point DNS to the last known good frontend host if apex cannot serve the rebuilt app.
- Keep API subdomain isolated; never redirect API traffic to the frontend.
